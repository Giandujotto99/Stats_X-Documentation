Setup Attributes

In this section you will create your Stats_X component hierarchy, attach it to your character, initialize your first attributes, and bind the first gameplay reactions.

What is an Attribute?

An attribute in Stats_X is a struct made of three floats. What those values represent depends on how you use the attribute. By default the system clamps values internally, for example Current stays between 0 and Max. If you want overflow behavior, enable bOverflows.

Before you start

Complete the Gameplay Tags setup first. Also, whenever this tutorial says rename, move to, or create a folder, treat those names as suggestions - feel free to adapt them to your own project structure.

Create the Stats Components

Create the master component

Go to your project's Content folder, not the plugin content folder. Right-click in the Content Browser, choose Blueprint Class, search for StatsX_StatsComponentBase, and create it.

Rename this blueprint BPC_StatsMaster.

Create the child component

Right-click BPC_StatsMaster and choose Create Child Blueprint Class.

Rename the child blueprint BPC_CharacterStats.

Attach the Component to Your Character

Open the character you want to use

Stats_X is not invasive, so you can use the character blueprint you already have in your project.

Add the stats component

In the Components panel, click Add, search for BPC_CharacterStats or whatever you named it, and add it to the character.

Your character now owns the Stats_X component. At this point it still does nothing by itself, and ticking is disabled.

Prepare the Initialization Event

Clean up the master blueprint

Open BPC_StatsMaster and remove the default events from the graph, leaving only Begin Play.

Create the server initialization event

Add a custom event called Server_InitializeAttributes and set it to Run On Server.

Initialize Your First Attributes

Open BPC_CharacterStats, go to the Functions panel, click Override, and choose Server_InitializeAttributes. That is where we will set up the first attributes.

Manual setup for learning

In a real project you will probably define these values through a Data Table or a Data Asset. For this tutorial we set them manually so the logic stays easy to follow.

Start with the first two nodes in Then 0

Add a Sequence node. In Then 0, place InitializeStatAttribute for the Level tag and leave Value and BaseValue at 0.

Next to it, add InitializeResourceAttribute for the Experience tag. Leave CurrentValue and MaxValue at 0, but set BaseValue to 100. We will use that later in the leveling formula.

Bind Begin Play and the Attribute Changed Event

Store a reference to your character

In BPC_CharacterStats, use the Begin Play event, call GetOwner, cast it to your character blueprint, and promote the result to a variable called CharRef.

Bind OnAttributeChanged

Call the OnAttributeChanged dispatcher. From the event input pin, create a Create Event node and choose Create a matching event from the dropdown.

Rename the event to OnAttributeChanged_Multicast. We are not splitting the bind logic with SwitchOnAuthority yet, so this bind will react on both server and client.

Initialize attributes on the authority branch

After binding OnAttributeChanged, add a SwitchOnAuthority node. On the Authority branch, call Server_InitializeAttributes.

Store a reference to the owning character during Begin Play.
Bind the dispatcher first, then initialize attributes on the authority branch.
Call Server_InitializeAttributes on the authority branch.
Important replication note

OnAttributeChanged_SomethingEvent nodes are not replicated RPCs by themselves. Replication happens internally before those events fire. That means you can split logic between server and client by deciding where you bind and react to the event.

Apply Movement Speed Immediately

Movement Speed is the first attribute you can see affecting gameplay right away, so it is a great first example of reacting to an attribute change.

Switch on the updated tag

In OnAttributeChanged_Multicast, call Switch on Tag List. This is a quality-of-life node; if you prefer the standard Switch on Tag, that works too.

Listen only for MovementSpeed

In the Details panel of the switch node, add the MovementSpeed tag and enable ExactMatch.

Apply the new max speed

From the MovementSpeed branch, set your character max speed using the event input parameter NewValue.Max.

Example of updating movement speed from the replicated attribute value.

UI Setup Options

At this point your attributes are initialized correctly on both server and client. For UI updates you can use either of these approaches:

  • Use OnAttributeChanged_ClientEvent, forward it through the Player Controller to your HUD, and update the widgets there.
  • From the UI, get the controlled pawn, fetch StatsX_StatsComponentBase with GetComponentByClass, bind to OnAttributeChanged, and update the text or numeric widgets directly.
UI tip

If you use the second approach, manually trigger OnAttributeChanged once right after the bind so the UI receives the initial values immediately.

You are done: your first attributes are initialized, replicated, and already driving gameplay.
