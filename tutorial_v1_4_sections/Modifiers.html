Modifiers

In this section you will understand what modifiers solve, how Stats_X recalculates them safely, and how to add or remove a modifier manually when you are not using Status Forge.

What is a modifier?

A modifier is a runtime adjustment applied to an attribute. In Stats_X, modifiers are designed to affect the derived Max value rather than permanently overwriting the original Base value.

Why Modifiers Exist

Modifiers solve a classic stacking problem. If you directly subtract and then re-add values, overlapping buffs and debuffs can easily restore the wrong amount and leave the attribute in an invalid state.

Example 
Movement Speed Base Value = 500

Enemy debuff: Additive = -300, Multiplicative = 1
New Max = (500 + (-300)) * 1 = 200

Snow debuff: Additive = -400, Multiplicative = 1
New Max = (500 + (-700)) * 1 = -200 -> clamped to 0

Enemy debuff ends
New Max = (500 + (-400)) * 1 = 100

Snow debuff ends
New Max = (500 + 0) * 1 = 500

That is the key benefit: removing one modifier never restores the wrong value, because Stats_X recalculates the final result from the full active stack every time (dirty flags for performance).

Recalculation formula

Stats_X recalculates modifiers with Max = (Base + Sum(Additive)) * Product(Multiplicative). This keeps overlapping effects stable and predictable.

Manual Modifiers vs Status Forge

Recommended workflow

You can create modifiers manually, but the recommended approach is to use Status Forge. It keeps your effect logic inside a single framework and works naturally with interceptors and status-driven cleanup.

Server-side only

Handle modifiers only on the server. Their state is replicated internally, so you should not manually apply them on both sides.

Add a Modifier Manually

If you want to create a modifier without Status Forge, use the AddModifier node from a Stats_X component.

Call AddModifier on the stats component

Do this from server-side gameplay logic, for example from your character, an ability, or a dedicated event that owns the effect.

Choose the target attribute

Modifiers are typically useful for values that influence the attribute Max. In this example we target Health.

Fill the modifier parameters

Set the source, owner, additive value, and multiplicative value according to the effect you want to create.

Parameter Name Example Value Meaning
AttributeTag Health The attribute affected by the modifier. In practice, modifiers are used to affect its Max value.
SourceTag Debuff.Arcane Identifies the source or category of the modifier. Useful for filtered cleanup such as removing all fire or arcane debuffs.
OwnerID 0 The creator or grouping ID of the modifier. In Status Forge this is handled automatically for you.
AdditiveValue -400 A flat change applied before multiplication.
MultiplicativeValue 1 A multiplier applied after additive values. For example, 0.7 means -30% and 1.3 means +30%.

Remove and Manage Modifiers

Once a modifier exists, you can remove it in different ways depending on the type of logic you need.

  • RemoveModifierByID for one exact modifier instance.
  • RemoveModifiersByOwner to remove everything created by the same owner or status.
  • RemoveModifiersBySource to remove a whole category such as a specific debuff source.
  • RemoveModifiersByAttribute to clear all modifiers affecting one attribute.
Common Blueprint nodes for removing or querying modifiers, plus additional utility helpers.
Best practice

If a modifier belongs to a temporary gameplay effect, give it a meaningful OwnerID or let Status Forge manage it for you. That makes cleanup far easier when the effect ends.
