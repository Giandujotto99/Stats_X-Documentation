Status Lifecycle

In this section you will learn how a status is cast, when it stays one-shot versus when it becomes active, how Stats_X tracks it for caster and target, and which dispatchers you should use to react to status changes.

What this section explains

A status is not only something you cast. It also has a runtime life: it can start, become active, update over time, expire, and notify both server and client through synchronized dispatchers.

Casting a Status

To cast a status you first need a valid Status Forge asset with a GameplayTag identifier and a compiled runtime definition.

Create the Status Forge asset

Build the status in Status Forge, assign its GameplayTag, and compile it so the runtime definition is generated.

Choose where to cast it from

If your actor has a Stats component, call CastStatusSync or CastStatusAsync on that component. If the actor does not have a Stats component, you can still cast the status from StatsX_WorldSubsystem.

Pass the correct casting parameters

Provide the status tag, caster, target, and any variable overrides needed by the status.

Multiplayer note

In multiplayer projects, cast statuses from the server side. Status execution and replication are designed around authoritative runtime control.

Parameter Meaning
StatusTag The GameplayTag identifier of the compiled Status Forge asset.
CasterActor The actor that is casting the status.
TargetActor The actor receiving the status. It can also be the same actor as the caster.
Overrides The set of variable overrides used to customize the status at cast time.

One-Shot vs Active Status

A status can exist in two runtime forms: One-Shot or Active.

One-Shot Status

A one-shot status behaves similarly to a function call. It is cast, executed, and resolved without needing to remain in the world as an ongoing active instance. It does not keep a lasting presence that needs to be tracked over time.

Typical examples are calculations, instant logic, triggers, or short procedural flows that do not need to stay alive after their work is done.

Active Status

An active status is a status instance that remains alive after its initial execution. It is stored in the world pool and, when available a Stats Component (not required), handle is referenced to the caster and target Stats components so it can continue ticking, waiting, or reacting to later events.

Important rule

All statuses begin as one-shot. A status becomes active only when execution reaches an async node such as LoopBehavior, UntilDeprecationBehavior, WaitForEvent, or other nodes that put the VM in a suspended state.

Where Active Statuses Are Tracked

Once a status becomes active, Stats_X tracks it in multiple synchronized places.

  • The world-level active status pool owned by UStatsX_WorldSubsystem.
  • The target component's ReceivedActiveStatuses container, if the target has a Stats component.
  • The caster component's CastedActiveStatuses container, if the caster has a Stats component.

The caster and target views are synchronized representations of the same active runtime status. In other words, the caster-side casted containers and the target-side received containers stay aligned with the same status handles and lifecycle events.

How a Status Ends

Active statuses are removed automatically when they expire, but they can also be removed manually by gameplay logic.

Status expires or is removed

The runtime instance reaches its natural end or is explicitly removed by code or graph logic.

References are cleaned up

The status is removed from the world pool and from the caster and target containers that were tracking it.

Registered interceptors are removed too

Any interceptors registered by that status are cleaned up automatically as part of the removal flow.

Status Dispatchers

Stats_X exposes synchronized dispatchers that help both server and client react to status changes. These are especially useful for gameplay reactions, UI refreshes, and client-side status tracking.

On Received Status Tag Changed

Fires when a status tag is added to the target's active received set, or when the last instance of that tag disappears from the received set.

Example: if Actor A receives Burning, the dispatcher fires with bAdded = true. If multiple Burning instances exist, it does not fire on every single removal - it fires again only when the last Burning instance is gone.

On Casted Status Tag Changed

Same idea as the previous dispatcher, but from the caster point of view. It fires when a status tag appears in or disappears from the caster's active casted set.

On Received Status Set / On Casted Status Set

These are lifecycle boundary events for a single active status instance, not whole-set refresh events. They fire when a status instance is added to or removed from the received or casted active set.

The payload is FStatusSetData, which contains an Action (Added or Removed) plus a Seed snapshot. That seed contains the client-facing status data needed to represent the instance in UI, including StatusID, StatusTag, ServerStartTimeSeconds, MaxDurationSeconds, TickIntervalSeconds, MaxIterations, StackCount, and Revision.

Use these dispatchers to add or remove a local entry for one specific status instance.

On Received Status Update / On Casted Status Update

These are delta update events for an already active status instance. They do not fire every frame, and they are not generic "something changed" notifications.

They are emitted only for non-deducible runtime updates, such as stack changes, refreshes, stack-change-plus-refresh, or runtime tuning. The payload is FStatusUpdateData, which contains StatusID, Revision, Reason, and a ChangedFields bitmask.

Only the fields marked inside ChangedFields are valid and should be applied on the client side.

UI utility mindset

The intended client-side workflow is to keep a local cache keyed by StatusID. When On...StatusSet fires with Added, store the incoming Seed. When it fires with Removed, remove that cached entry. When On...StatusUpdate fires, patch the cached data of that same StatusID using only the fields flagged in ChangedFields.

Client-side flow: cache lifecycle seed data per status instance, patch it with delta updates, and drive the UI from that local state.
Client access tip

Do not expect a full replicated status-instance object from these dispatchers. The current system replicates lifecycle seed snapshots and delta update events. For timers, use the cached FStatusSeedData together with GetStatusInstanceRemainingTime. For update payloads, use helper checks such as StatusUpdateHasStackCount, StatusUpdateHasStartTime, StatusUpdateHasMaxDuration, StatusUpdateHasTickInterval, and StatusUpdateHasMaxIterations.

You are done: you now know how statuses are cast, when they become active, how they are tracked, and how to build client-side status UI correctly using lifecycle seed events plus delta updates.
