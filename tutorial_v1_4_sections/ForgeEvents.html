Forge Events

Forge Events are the bridge between Blueprint gameplay and a running status. Instead of guessing the right moment with a Delay, a status can wait for a real gameplay signal such as an animation notify, a projectile callback, an interaction, or any other Blueprint-driven event.

What Forge Events do

WaitForEvent and WaitForEvents let a status pause and listen. Later, SendForgeEvent wakes the waiting status and resumes the correct event branch.

When to use them

Forge Events are perfect when status execution must react to something external: a montage notify, a weapon release frame, a projectile hit, a combo input window, or a world interaction.

The Example We Will Build

In this tutorial we build a practical Blueprint flow: the player starts a casting status, the status waits for a montage notify, and when the notify fires the status resumes and releases the projectile.

Blueprint Flow 
Player input
-> Cast Status
-> Status plays montage / enters WaitForEvents
-> Montage notify fires in Blueprint
-> Blueprint calls SendForgeEvent
-> Status resumes in OnEventTriggered
-> Projectile or gameplay effect is released

Why This Is Better Than Delay

  • A montage can change length, but the notify still marks the correct frame.
  • The status resumes only when the real event happens, not when a guessed timer expires.
  • The same pattern works for projectiles, interactions, combo windows, and many other Blueprint events.

Create the Event Tag

Create a GameplayTag for the event

Add a new tag under the event hierarchy, for example StatsX.Event.Gameplay.Fireball.Release.

Name it after the real gameplay moment

Good event tags describe when something happened, not what you hope will happen next. Think in terms of Release, Impact, Combo.Continue, Interaction.Accepted, and similar signals.

Build the Waiting Status

Now build or open the status that should wait for the external signal. In this example it is the casting status that should release the projectile only after the animation reaches the notify.

Choose the waiting node for the graph

If you only need one event, WaitForEvent is enough. In this example we will use WaitForEvents because it gives us more control, such as per-event trigger limits and max waiting time.

Configure it with the event tag you created, such as StatsX.Event.Gameplay.Fireball.Release.

Use the immediate branch for setup

The unnamed branch of WaitForEvent, or the Now branch of WaitForEvents, runs immediately in the same frame. Use it for setup work such as starting a montage, spawning telegraph VFX, or preparing state before the event arrives.

Use the event handler for the real reaction

In WaitForEvent, put the gameplay action you actually want to delay in the Event Triggered branch. In WaitForEvents, define the event entries in the Events parameter, then create one OnEventTriggered node for each event you want to handle.
For a fireball, this is where you would release the projectile, cast the hit status, or continue the combo.

Create the OnEventTriggered handlers

If you are using WaitForEvents, the output is the one that runs instantly before the status enters suspension and starts waiting for events.
To create the branch for a specific event, add a new OnEventTriggered node. Assign to it one of the events you configured in WaitForEvents, then define how many times it can trigger and the max waiting time. In this example the waiting time is 2 seconds, so if the montage is interrupted or something goes wrong, the status does not wait forever and can auto terminate.
You need one OnEventTriggered node for each event tag you want to react to.

Waiting Flow Part When it runs Typical use
WaitForEvent Immediate / WaitForEvents Now Right away, in the same frame Play montage, setup visuals, start anticipation logic
WaitForEvent Event Triggered / WaitForEvents OnEventTriggered Later, when the matching event is sent Release projectile, apply hit logic, continue combo flow
Important timing rule

The event branch does not resume instantly in the same frame as SendForgeEvent. The event is queued and the status resumes on the next subsystem tick.

Default listener behavior

WaitForEvent registers an infinite listener by default. This means the same waiting status can react again and again to the same event until the status ends. If you want a single-use reaction, make the status end after the first trigger or use WaitForEvents with a trigger budget.

Send the Event from Blueprint

Once the status is waiting, you need to send the event at the correct gameplay moment.

Choose the Blueprint that knows the moment has happened

For this example, the best source is the animation notify inside the Montage. In other cases, it could be the projectile Blueprint, an interactable actor, or a character event graph.
Go to the Content Browser, right-click, create a new Blueprint class, and choose AnimNotify as parent class.
Open it, add a new variable of type GameplayTag, name it EventTag and click the eye icon to make it InstanceEditable.
Now go to your animation montage and add this notify at the frame where you want to send the event.
In the details panel of the notify, you can now choose the tag you want to send. It must match the tag used by the waiting node, or one of the tags configured inside WaitForEvents.

Call SendForgeEvent with the same EventTag

When the notify or Blueprint event fires, call SendForgeEvent using the exact same tag the status is waiting for. In the notify, override Received_Notify and call SendForgeEvent there. In this kind of Blueprint, the Blueprint Library version is often the best choice because you can pass the scope actor explicitly.

Make sure the scope actor matches

A waiting status listens with an actor scope. Internally, the waiting node uses the target actor first, and if there is no target it falls back to the caster actor. If you send the event from the wrong scope actor, the waiting status will not resume.

In an AnimNotify setup, this usually means passing the mesh owner or character owner as the scope actor.

Three Practical Ways to Send It

Method Best use case Scope behavior
Stats Component - SendForgeEvent When the actor already owns a Stats component Uses the component owner automatically as the scope actor
Blueprint Library - SendForgeEvent Anim Blueprints, notifies, widgets, or any other Blueprint context You pass the scope actor explicitly
WorldSubsystem - SendForgeEvent More advanced setups that already work with the subsystem directly You pass the scope actor explicitly
Easy rule for self-cast statuses

If the status is cast by the same actor that should receive the event, calling SendForgeEvent from that actor's Stats component is usually the easiest and safest option.

Send Payload with the Event

Forge Events can also carry data. The payload uses the same override struct you already know from variable overrides: FForgeVariableOverride.

Create variables in the waiting status

Add the variables the event should fill. A very good convention is to keep them under the dedicated subtree StatsX.Variable.Payload.*, for example StatsX.Variable.Payload.ChargedDamage or StatsX.Variable.Payload.TargetLocation.

Build the payload array in Blueprint

Use the Blueprint helper nodes such as MakeFloatOverride, MakeGameplayTagOverride, MakeVectorOverride, and the other override factories to assemble the payload.

Read the variables directly after resume

When the waiting status resumes, the payload has already been applied to the status variables. That means the Event Triggered branch can use the new values immediately.

Variable in Status Payload sent from Blueprint What the handler can do
StatsX.Variable.Payload.ChargedDamage MakeFloatOverride(..., 125.0) Scale damage based on the real charge amount
StatsX.Variable.Payload.ReleaseSocket MakeNameOverride(...) Choose which socket or release point to use
StatsX.Variable.Payload.TargetLocation MakeVectorOverride(...) Resume the event branch with a precise world-space target
Keep payload simple

Scalars, tags, object references, class references, and common value types work well as payload. For Blueprint tutorials and gameplay iteration, keep event payloads small and focused instead of trying to send large structures or collection-heavy data.

Use WaitForEvents for More Advanced Flows

Use WaitForEvents when one status should react to multiple possible signals instead of only one.

This node is especially useful for combo windows, interrupt logic, reaction branches, or any gameplay situation where the next step depends on which event arrives first.

Event Typical handler MaxTriggers MaxWaitingTime
StatsX.Event.Gameplay.Combo.Continue Go to the next combo branch 1 0.35
StatsX.Event.Gameplay.Combo.Cancel Abort the combo flow 1 0.35
StatsX.Event.Gameplay.Combo.Bonus Apply an optional extra branch -1 -1
How WaitForEvents differs from WaitForEvent

WaitForEvent is the simplest persistent listener: one tag, infinite triggers, no timeout. WaitForEvents adds per-event handler branches, per-event trigger limits, and per-event waiting times.

Common Mistakes

  • Using the right event tag but sending it from the wrong scope actor.
  • Expecting the event branch to run in the same frame instead of the next tick.
  • Using Delay for animation-driven moments that already have a notify.
  • Forgetting that WaitForEvent keeps listening forever unless the status ends.
  • Trying to solve a complex multi-branch gameplay flow with one single event when WaitForEvents would be clearer.

Final Mental Model

Think of Forge Events like this: Blueprints tell the world when something real happened, and Status Forge decides what that means for gameplay.

Once you start using this pattern, your statuses stop depending on guessed delays and begin reacting to real gameplay timing. That makes your logic cleaner, easier to reuse, and much more reliable.
