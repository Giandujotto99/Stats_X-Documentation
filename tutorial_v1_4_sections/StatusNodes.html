Status Nodes

Nodes are the building blocks of Status Forge. They are the visual pieces that define how a status behaves, interacts with attributes, reacts to events, and evolves over time.

How to think about a node

Each node is an authoring tool in the editor, but at compile time it becomes a real instruction handled by the Forge runtime. That is why the system feels visual while still staying performant.

Every built-in node is documented, and new updates will continue expanding the available toolset. If you ever need something more specific, Stats_X also lets you extend the system with your own logic.

Built-in Nodes and Custom Nodes

Status Forge already gives you a wide set of nodes, but you are not limited to the built-in set.

  • Use the built-in nodes for standard gameplay flow, checks, math, modifiers, statuses, and events.
  • If you need fully native custom behavior, you can create your own nodes in C++.
  • If you work in Blueprints, you can extend the system through Custom Actions and Custom Checks.
Custom logic lets you extend Status Forge without changing the plugin workflow.

Create Your Own Custom Logic in Blueprints

Create a new Blueprint class

In the Content Browser, create a new Blueprint class and choose either ForgeCustomAction or ForgeCustomCheck depending on the type of node you want to build.

Write the logic inside the Blueprint

Implement the gameplay behavior or condition you need, exactly like you would for any other Blueprint-based extension.

Expose editable variables

Create instance-editable variables in the Blueprint. Those variables become pins or configurable parameters in Status Forge, so the node can be customized directly from the graph.

Blueprint-friendly extension path

This is the easiest way for Blueprint users to create reusable node behavior without modifying the plugin source code.

Connection Rules

There is one especially important rule to remember when connecting nodes in Status Forge: pay attention to blue nodes that work across time.

Blue timed nodes can cross frames

If a blue node has a duration or any time-based behavior, it means the execution can continue in a later frame. Data coming from nodes before it may no longer be available in the way you expect when the graph resumes after time has passed.

In practice, this means you should not assume temporary data from a node before a time-based node will still be safely usable in nodes after it. If the graph spans multiple frames, the context has changed.

Bad connection pattern

A node produces data, then a time-based blue node waits or runs across multiple frames, and finally a later node tries to use that earlier data as if everything still belonged to the same moment.

Incorrect pattern: data is treated as if execution never left the same frame.
Good connection pattern

Restructure the graph so the logic after the timed node uses data that is still valid for that later frame, or rebuild the needed data after the delay instead of depending on the old transient state.

Correct pattern: the logic is rebuilt in a way that respects multi-frame execution.
Practical mindset

If a node waits, loops with time, or otherwise spans multiple frames, think of the logic after it as a later moment in gameplay, not as a direct continuation of the same instant.

Once you understand this rule, building larger statuses becomes much safer. You can combine built-in nodes, custom logic, and time-based behavior without creating subtle execution mistakes.

Avoid Deep Async Chains

Do not chain too many time slices

When too many async nodes are chained together, they start competing over timing, suspension, resume flow, and active-status ownership. In simple cases it may still work, but in more complex graphs you will often see wrong timing, missed events, or logic that does not trigger as expected.

The safest solution is simple: if the logic becomes too asynchronous, split it into multiple statuses with clear responsibilities instead of forcing everything into one deep chain. This keeps the execution easier to reason about and makes debugging much easier too.
