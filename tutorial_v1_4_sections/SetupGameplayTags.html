Setup GameplayTags

Before creating attributes, statuses, and cooldown logic, take a moment to design your GameplayTag hierarchy. A clean hierarchy will make the rest of the project much easier to scale.

What this section is really about

You do not need to plan every single tag for the entire game today. The goal is to create a strong, readable structure so future tags can fit naturally instead of becoming chaos over time.

Good habit

Write your hierarchy down before you start. A simple document, Google Sheet, or Miro board is often enough to avoid renaming and reorganizing everything later.

Plan the Main Tag Families First

Start by thinking about the broad categories your game needs. For a typical Stats_X setup, the most useful families are usually:

  • Attributes - stats, resources, resistances, stacks, progression.
  • Status - skills, environment effects, gameplay states, cooldown entries.
  • Cooldown branches - separate when you know some systems should affect only a subset.

A structure like this is useful because it lets you target both broad groups and precise leaf tags. For example, you can affect all Vital attributes, only Resistances, or just one specific value such as MovementSpeed.

Example Status Hierarchy

Status tags benefit from the same idea: start broad, then specialize based on delivery type, element, system, or effect family.

Why Hierarchy Matters

A strong hierarchy is not just nice to look at. It gives you design control later.

  • An item may reduce all skill cooldowns by 20% after a kill.
  • A passive may reduce all ranged or projectile damage by 10%.
  • A debuff may amplify all crowd control statuses on the target.
  • A silence effect may block only spell-related casts without touching everything else.

If your tags are flat, those kinds of effects become painful to express. If your hierarchy is clean, they are easy to target.

Do not overbuild too early

Keep the hierarchy strong, but do not try to predict your entire game in microscopic detail. Plan the families and the naming style first, then expand naturally as the project grows.

Add the Tags in DefaultGameplayTags.ini

You can create tags manually in the editor, but once the list starts growing, editing the config file is much faster.

Close the project after installing Stats_X

This is the safest moment to bulk-edit your tag list without Unreal keeping the file open while you work.

Open your config file

Navigate to YourProject/Config/DefaultGameplayTags.ini.

Add your hierarchy entries

Write the tags directly in the file, save it, then reopen the project.

DefaultGameplayTags.ini 
[/Script/GameplayTags.GameplayTagsList]
+GameplayTagList=(Tag="Attributes.Stat.Progression.Level",DevComment="")
+GameplayTagList=(Tag="Attributes.Stat.Primary.Strength",DevComment="")
+GameplayTagList=(Tag="Attributes.Resource.Vital.Health",DevComment="")
+GameplayTagList=(Tag="Attributes.Stat.Resistance.Arcane",DevComment="")
+GameplayTagList=(Tag="Status.Skill.Projectile.Arcane.FireballCast",DevComment="")
+GameplayTagList=(Tag="Status.Skill.Projectile.Arcane.FireballEffect",DevComment="")
+GameplayTagList=(Tag="Status.Skill.DirectTarget.Arcane.ManaShield",DevComment="")
+GameplayTagList=(Tag="Status.Cooldown.Skill.FireballCD",DevComment="")
Simple naming rule

Go from generic to specific. If a tag reads naturally from left to right, the hierarchy is usually doing its job correctly.

These are only examples. First follow the tutorial to understand how the plugin works, then apply this new knowledge to your own project and build a strong tag structure that fits your game.
