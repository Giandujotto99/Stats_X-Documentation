Status Forge

Status Forge is the heart of Stats_X authoring. In this section you will understand what it is, what a status really becomes at runtime, and how the editor helps you build reusable gameplay logic visually.

What is Status Forge?

Status Forge is a visual editor that lets you build complex Status Definition assets in a Blueprint-like workflow. Each node is compiled into instructions executed by the Forge virtual machine, so the final runtime status stays fast and CPU-friendly.

Editor-only authoring, runtime output

The Status Forge asset itself is editor-only. The real gameplay asset is the compiled data asset it generates. Every time you save and compile, Stats_X updates that runtime definition for you.

What Is a Status?

A status is a compiled gameplay definition stored in a data asset as an optimized list of instructions. In practice, a status is simply something that happens.

  • A spell, skill, passive, aura, or timed gameplay effect.
  • A world action, such as opening a door, spawning an enemy, or entering a boss arena.

If you know GAS, you can think of a Stats_X status as covering the role of both a Gameplay Ability and a Gameplay Effect, but with a workflow that is faster to author, easier to share, and cheaper to execute.

Understand Status Core Data

The core data section defines the fundamental rules of a status. This is the part that decides how the status is identified, how it stacks, and how it behaves when cast repeatedly.

Field Meaning
StatusTag The GameplayTag used to cast, refresh, update, or remove this status.
Description A designer-friendly tooltip that documents the purpose of the status.
Max Stacks The maximum number of stacks. A value of -1 means infinite stacks.
Stack Per Caster If enabled, different casters maintain separate stack groups on the same target.
ExecuteOnEndForNaturalTermination If enabled, the OnEnd logic runs when the status ends (can be overwritten runtime).

Stacking Policies

Policy Behavior
Aggregate Adds a stack if the same status is already active on the target.
Aggregate and Refresh Adds a stack and also resets the duration.
Refresh Only Does not add stacks, but refreshes the duration.
Override Removes the old status and replaces it with the new cast.
Reject Ignores the new cast if the same status is already active.
Independent Always allows the cast, even if the same tag is already active.
Why Stack Per Caster matters

If two different actors cast the same aggregate status on one target, enabling Stack Per Caster keeps those stacks separate. That makes ownership and cleanup much easier to reason about.

Compile and Runtime Output

Once the graph and core data are ready, click Compile. This generates or updates the runtime data asset used in-game.

Compile turns the visual graph into the runtime status definition.

The generated data asset is automatically assigned to the DA Status Definition reference, and it is also registered inside the DA_ImmutableGameStatuses asset created during installation.

Important workflow note

Compiling again does not create duplicate runtime assets. It simply updates the same generated status definition. That means you can iterate safely without cluttering your project.

The generated data asset reference and compile timestamp.

Node Selection and Editing

Nodes can be spawned in the graph by right-clicking, exactly like in Blueprints. If you prefer, you can also use the palette panel on the left. If you closed it by mistake, reopen it from Window -> Palette.

Use either the graph context menu or the palette to add nodes.

When you select a node, the details panel exposes all of its editable parameters. The node itself also shows a concise summary of the most relevant information you configured.

Selected nodes expose their parameters through the details panel.

Import and Share Statuses

One of the strongest ideas behind Status Forge is portability. When you save and compile a forge asset, Stats_X can export a JSON representation that recreates the exact same forge structure in another project: nodes, connections, variables, configuration, and the overall editable logic.

This is important because the JSON is not just a summary or a final compiled result. It is a portable version of the authored status itself. When you import it, you are not receiving only an effect that works - you are receiving the full starting point for further design work.

What the JSON preserves

The imported JSON recreates the same status authoring structure, so you can inspect it, edit it, extend it, and make it your own instead of rebuilding the whole graph manually from zero.

This is what makes the Library truly powerful: a status can be created once, shared as JSON, imported in a few seconds, and then adapted to a completely different project. You may only need to change some values, swap a tag, replace a calculation, or add your own branch of logic - but the heavy work is already done.

In practice, this turns statuses into reusable building blocks. Instead of spending an hour recreating the same graph, you can open the Library, choose the closest status, import it, study how it was built, customize it, and move forward immediately. That is not only a convenience feature - it is a multiplier for iteration speed, knowledge sharing, and community growth.

Import JSON recreates the same forge asset in seconds.
Community value

The more reusable statuses the community shares, the stronger the Library becomes, and the faster new gameplay systems can be built.

Forge Variables

Forge variables make statuses flexible. They let you define data that can be overridden at cast time or changed inside the status logic itself.

  • StatsX.Variable.* is for real forge variables that you override or edit during execution.
  • StatsX.Variable.Payload.* is for data coming from payloads.

If you know GAS, think of this as the bridge that allows nodes such as WaitForEvent and WaitForEvents to receive payload data from elsewhere in your project through SendForgeEvent.

Forge variables let one status definition adapt to many different use cases.

Why Status Forge Matters

Status Forge is more than just an editor. It is the system that turns reusable gameplay logic into something you can author visually, compile into efficient runtime data, export to JSON, import again in seconds, and keep growing over time.

This is the most powerful feature in Stats_X, and it is the foundation for the next sections where the nodes and connection rules become even more important.
