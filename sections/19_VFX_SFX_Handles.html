1. Overview

Stats_X manages spawned Niagara particle systems and persistent audio components through a handle-based registry hosted on UStatsX_WorldSubsystem. Every VFX or SFX spawned by the ForgeVM receives a unique int64 handle. The subsystem maps that handle to the spawned component, enabling later operations — update, stop, or destroy — to locate the component by handle alone, even across the network.

The handle layer sits between the ForgeVM opcode handlers and the Cosmetic Cue bus, ensuring that:

  1. The server can reference effects by handle in cosmetic cues without holding a UObject pointer.
  2. Clients can independently spawn and track their own component instances.
  3. Stale entries are lazily purged so the registries stay bounded.

2. Handle Allocation

Handles are unique identifiers assigned automatically when spawning VFX/SFX. Both VFX and SFX use independent allocators. Handle 0 is reserved as the invalid sentinel; all API entry points early-return on invalid handles.

Property VFX SFX
Component type UNiagaraComponent UAudioComponent
Handle type int64 int64
Invalid sentinel 0 0

3. VFX Handle Registry API

All methods live on UStatsX_WorldSubsystem.

AllocateCosmeticVFXHandle

C++ 
int64 AllocateCosmeticVFXHandle()

Returns the next unique VFX handle.

RegisterLocalVFXComponent

C++ 
void RegisterLocalVFXComponent(int64 VFXHandle, UNiagaraComponent* Component)

Maps a handle to its spawned Niagara component. If Component is invalid, the entry is removed instead (defensive cleanup).

FindLocalVFXComponent

C++ 
UNiagaraComponent* FindLocalVFXComponent(int64 VFXHandle)

Returns the component for the given handle, or nullptr if not found or stale.

UnregisterLocalVFXComponent

C++ 
void UnregisterLocalVFXComponent(int64 VFXHandle)

Removes the handle from the registry unconditionally.

4. SFX Handle Registry API

The SFX registry is structurally identical to the VFX registry, substituting UAudioComponent for UNiagaraComponent.

ℹ️ Note

SFX handles are only allocated when EStatsXSFXPlaybackPolicy == Persistent. Fire-and-forget sounds receive handle 0 and are not tracked.

5. ForgeVM Opcode Handlers

Six opcodes interact with the handle registries for spawning, updating, and stopping VFX/SFX.

OP_SpawnVFX

Spawns a Niagara system at a world location and returns a handle for later control. The node accepts a Niagara system asset, transform, transform space, net delivery mode, and pool policy. On the server, a cosmetic cue is emitted so clients can spawn the effect locally.

OP_SpawnVFXAttachedtoActor

Spawns a Niagara system attached to an actor socket. In addition to the standard VFX parameters, this node accepts a parent actor, socket name, and attach rule. If the named socket is not found, the handler logs a warning and falls back to the root component.

OP_StopVFX

Stops and deactivates a previously spawned Niagara system by handle. Accepts an optional blend-out time — if zero or near-zero, the effect is deactivated immediately; otherwise Niagara blends out naturally.

OP_UpdateVFX

Updates the world transform of a live Niagara system identified by handle.

OP_SpawnSFX

Spawns an audio source at a world location. Supports two playback policies:

Policy Behavior Handle Tracked
FireAndForget (or Auto) One-shot playback 0 No
Persistent Tracked audio component allocated Yes

Audio parameters (volume, pitch, start time) are validated and reset to safe defaults if invalid.

OP_StopSFX

Stops a persistent audio component by handle. Accepts an optional fade-out time — if zero or near-zero, the sound stops immediately; otherwise it fades out over the specified duration.

7. Handle Lifecycle

The VFX/SFX handle lifecycle follows a simple pattern on both server and client:

  1. Allocate — a unique handle is generated when spawning an effect.
  2. Register — the spawned component is associated with the handle in the local registry.
  3. Use — subsequent operations (update, stop) reference the effect by handle.
  4. Unregister — when the effect is stopped or destroyed, the handle is removed from the registry.

On the server, cosmetic cues carry the handle so that clients can independently spawn and register their own local component instances using the same handle identifier.

SFX lifecycle (Persistent) follows the same flow. Fire-and-forget sounds bypass the handle registry entirely.

8. Network Execution Model

Context Local spawn Cue emitted Handle registered
Dedicated server No Yes No
Listen server Yes Yes Yes
Standalone Yes No Yes
Client (from cue) Yes No Yes

On a dedicated server, VFX/SFX opcodes only emit cosmetic cues — no Niagara or audio components are spawned. Clients receive the cues through the NetCueRelay and execute the spawn locally, registering the component in their own local handle registry.
