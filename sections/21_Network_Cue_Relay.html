1. Purpose

This document is the actor-level reference for AStatsX_NetCueRelay — the lightweight AInfo subclass that transports cosmetic cues from the server to clients. While 18 — Cosmetic Cue Routing covers the routing pipeline end-to-end, this document focuses on:

  • The relay actor’s configuration and modes.
  • Montage lifecycle integration.
  • The EStatsXCosmeticCueType::Custom extension point.
  • Built-in cue execution summary.

3. Configuration Modes

3.1 Route-Actor Relay

The relay inherits relevancy from a designated route actor. Clients that can see the route actor automatically receive the relay and its cues. Clients outside relevancy range receive nothing — zero wasted bandwidth. The relay is configured by setting the route actor as its owner and using owner-based relevancy.

3.2 Global Relay

Always relevant to all connected clients. Used for cues with no actor affinity or as a fallback when the route-actor relay cap is exceeded. The global relay has no owner and is marked as always relevant.

The conceptual difference: route-actor relays are bandwidth-efficient because they only replicate to clients that can see the associated actor, while global relays broadcast to everyone regardless of distance or relevancy.

4. State and Burst Cue Delivery

The relay supports two delivery channels:

Channel Reliability Join-in-Progress Use Case
State ReliableReliable, delta-compressedSafe (new clients get full state)Persistent effects, montages
Burst UnreliableUnreliable multicast RPCNot safe (fire-and-forget)One-shot VFX/SFX

State cues are replicated with latest-value-wins semantics — if the same effect is updated multiple times before replication occurs, only the latest value is sent. New clients joining mid-session receive the full state array, making the system join-in-progress safe for state cues.

5. Client Reception

When cues arrive on the client, they pass through a three-stage dispatch pipeline:

Stage Action
1. Built-in executionImmediate local playback (montage, VFX, SFX)
2. Native delegateC++ delegate for custom consumers
3. Bus re-injectionRe-inject into local cue bus for Blueprint/sink consumers

6. Montage Lifecycle Integration

Montage playback integrates with the status lifecycle for blend and completion callbacks. When a montage is played via OP_PlayMontage, a helper object is created to bridge montage animation delegates with the ForgeVM execution context.

Each montage signal (BlendIn, BlendOut, Interrupted, Cancelled, Completed) maps to a distinct continuation point in the status definition’s instruction stream, allowing designers to branch logic per montage event.

6.1 Status Termination

When a status that owns a playing montage is removed, the system can optionally auto-stop the montage (controlled by the bStopOnStatusEnds flag). If enabled:

  • A StopMontage cosmetic cue is emitted so clients stop the montage as well.
  • The montage is stopped locally with a brief blend-out.
  • The StopMontage cue overwrites the PlayMontage state in the replicated state array, keeping the system join-in-progress consistent.

6.2 Montage Signal Flow

Structure 
OP_PlayMontage → create Helper → bind anim delegates
                                     ↓
                         OnMontageBlendedIn  → signal BlendIn
                         OnMontageBlendingOut → signal BlendOut/Interrupted
                         OnMontageEnded       → signal Completed/Cancelled
                                     ↓
                         VM resumes at signal-specific continuation point

7. Custom Cue Extension Point

7.1 EStatsXCosmeticCueType::Custom

The Custom cue type is not handled by the built-in cue executor (falls through as a no-op). It exists as an extension point for game-specific cues.

7.2 OnClientCueReceived Delegate

C++ 
FOnStatsXNetRelayCueReceivedNative& OnClientCueReceived();

Broadcast after built-in cue execution for every cue (not just Custom type). Custom cues pass through built-in execution as a no-op, then hit this delegate where game code can interpret the payload.

7.3 Bus Sink Delegate

C++ 
FDelegateHandle RegisterCosmeticCueSink(
    const FOnStatsXCosmeticCueRoutedNative::FDelegate& Delegate);

Server-side sink registration for custom routing logic.

7.4 Extension Pattern

C++ 
// Game code: register a custom sink for game-specific cue routing
FDelegateHandle Handle = Subsystem->RegisterCosmeticCueSink(
    FOnStatsXCosmeticCueRoutedNative::FDelegate::CreateUObject(
        this, &UMyComponent::OnCosmeticCueRouted));

// Or listen on a relay for client-side custom cues
Relay->OnClientCueReceived().AddUObject(
    this, &UMyComponent::OnClientCueReceived);

8. Relay Lifecycle Management

8.1 Creation

Relays are created on demand via GetOrCreateNetCueRelay(AActor* RouteActor) on the world subsystem:

  • If RouteActor is null or not replicated, a global relay is created or reused.
  • If RouteActor is valid and replicated, a route-actor relay is created or reused from a per-actor cache.
  • If the route-actor relay cap is exceeded, the system falls back to the global relay.

8.2 Fallback Chain

Two-level fallback: route-specific → global. A cue always finds a relay unless the world is shutting down.

8.3 Destruction

All relays are destroyed during world subsystem teardown. Both the global relay and all per-route-actor relays are cleaned up.

10. Design Decisions

Decision Rationale
Single actor class, two modesRoute-actor and global relays share code; mode is runtime configuration, not inheritance
Owner relevancy for route relaysLeverages UE5’s built-in relevancy without custom IsNetRelevantFor
Custom cue type as no-opZero overhead for built-in dispatch; game code opts in via delegate
Three-stage client dispatchBuilt-in execution → native delegate → bus re-inject covers all consumer patterns
bStopOnStatusEndsDesigner-controlled: automatic cleanup vs manual lifecycle management
