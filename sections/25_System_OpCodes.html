1. Overview

Every ForgeVM instruction is identified by an opcode. The opcode space is partitioned into three categories:

Category Purpose
System OpCodes VM control flow — reserved ranges for internal use.
Built-In Node OpCodes Shipped with the plugin — standard node operations.
Custom OpCodes Auto-generated from .toon files — project-defined extensions.

The system reserves ranges for internal opcodes while allowing custom extensions. Built-in and custom opcodes are dispatched to registered handler functions. Custom opcodes are regenerated on editor startup.

2. Execution Results

All opcode handlers return one of the following:

Result Meaning
Continue Succeeded, advance to next instruction.
Completed Execution finished successfully.
Suspended Async operation started, will resume later.
Error Runtime error occurred.
Aborted Execution manually aborted.

3. System OpCodes

System opcodes handle VM control flow. These are internal to the ForgeVM and not directly used by end users.

Name Description

OP_Nop No operation.
OP_BeginBlock Start of a flow control block.
OP_EndBlock End of a flow control block.
OP_ContinueData Continuation data slot.
OP_BeginEventBlock Start of an event/function block.
OP_EndEventBlock End of an event/function block.
OP_CallFunction Call a function block.

5. Custom OpCodes

Custom opcodes are auto-generated from Custom Node Definition files (.toon) and regenerated on editor startup.

Property Value
Source .toon files in the project
Generation Automatic on editor startup

The base installation ships with an empty custom opcode namespace. Custom node definitions added to the project are assigned sequential opcode values in a reserved range.
