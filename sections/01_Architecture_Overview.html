Design Philosophy

Stats_X is built around four core principles:

Principle Implementation
Zero-allocation hot path Execution context is reused across invocations; status instances are pooled to avoid runtime allocations
Cache-friendly design Instructions and runtime data are laid out for optimal cache performance
Handler-driven lifecycle The VM executes bytecode inline — node handlers decide if/when to promote an execution to a persistent instance
Visual authoring, native performance StatusForge graph editor compiles to compact bytecode executed by the ForgeVM at native speed

The system deliberately avoids GAS patterns that introduce allocation pressure (per-ability instancing, deep UObject hierarchies, Gameplay Effect pooling) and instead adopts a bytecode VM approach where status definitions are data assets compiled to instructions, and the runtime uses an efficient dispatch mechanism for registered handlers.

Runtime Layer Map

Layer 1 — Data

Immutable, compiled assets produced by the StatusForge editor.

Structure Role
UPDA_StatusDefinition Compiled status asset containing the instruction array, typed literal pools, variable declarations, and function definitions
FStatusInstruction Single VM instruction using a fixed-width format optimized for cache performance. Supports continuation slots for large payloads
Literal Pools Type-safe constant pools (floats, ints, bools, tags, objects, vectors, transforms, strings, structs, etc.) indexed from instruction payloads
OpCode Tables System OpCodes, Built-In Node OpCodes, and Custom Project OpCodes (10000+)

Layer 2 — Execution

The virtual machine and its runtime state.

Component Type Role
FForgeVM Singleton Bytecode interpreter. Uses an efficient dispatch mechanism to execute instructions with no per-execution allocation
FForgeVMContext Struct (reused) Transient execution state holding scratch memory, variable storage, object references, and function call frames. Reused across executions
FStatusInstance Struct (pooled) Persistent state for suspended/active effects: timing, iteration count, stack count, variables, and saved execution position
FStatusInstancePool Struct Pre-allocated pool of FStatusInstance. Handle-based access via StatusID. Owned by UStatsX_WorldSubsystem

Layer 3 — Component

Per-actor gameplay state.

Component Role
UStatsX_StatsComponentBase Actor component holding attributes, modifiers, thresholds, local interceptors, and replicated data. Entry point for ApplyStatus()
FStatAttribute Value triplet: Current / Max / Base. Clamping, overflow toggle, replication flag
FStatModifier Additive + Multiplicative modifier. Identified by InstanceID, grouped by OwnerID (batch removal)
Local Interceptors Per-component interceptor registrations with Caster/Target scope filtering
Replication Attributes and modifiers replicate via FFastArraySerializer for delta-only network updates

Layer 4 — Orchestration

World-level coordination and lifecycle management.

Component Role
UStatsX_GameInstanceSubsystem Global StatusID generation. Holds the DA_ImmutableGameStatuses registry for tag-based status lookup
UStatsX_WorldSubsystem Central authority: executes statuses, owns the instance pool, ticks instances, manages stacking policies, routes cosmetic cues, manages global interceptors, runs the Forge Event System

Layer 5 — Presentation

Visual/audio feedback and network transport for cosmetic state.

Component Role
Cosmetic Cue Bus Enqueue / flush pipeline for FStatsXCosmeticCue (montage, VFX, SFX, custom). Deduplication via stream key
VFX / SFX Handle Registry Monotonic handle allocation, local Niagara/Audio component tracking
AStatsX_NetCueRelay Per-route-actor replicated actor. Transports cosmetic cues to clients via state-reliable or burst-unreliable channels

Execution Model

Inline Execution (default)

Every status begins as an inline execution — the VM runs bytecode synchronously from start to end (or until a suspend point). No pool instance is created unless a handler explicitly requests it.

Flow 
WorldSubsystem.ExecuteStatus(StatusID, Definition, Caster, Target)
  │
  ├─ Apply stacking policy (Aggregate, Refresh, Override, Reject, Independent)
  ├─ ForgeVM.ExecuteStatus()
  │    ├─ Setup execution context (Definition, Caster, Target, timing)
  │    ├─ Apply variable overrides (if any)
  │    └─ Run instruction loop until terminal state
  │
  └─ Result: Completed | Suspended | Error

Instance Resumption

Suspended instances are ticked by the WorldSubsystem. When their timer expires or a Forge Event fires, the VM resumes execution from the saved position:

Flow 
WorldSubsystem.Tick(DeltaTime)
  │
  ├─ Process pending async resumes (montage callbacks, etc.)
  ├─ Tick active instances (timer-based resumption)
  ├─ Process Forge Event triggers
  ├─ Flush pending status updates (coalesced delta replication)
  └─ Flush cosmetic cue bus (route and dispatch cues)

Catch-Up Mechanism

If a status falls behind (frame hitches), the WorldSubsystem executes multiple iterations per frame, capped by MaxExecutionsPerFrame (default: 100). Optional time-drift compensation ensures precise scheduling over long durations.

Data Flow

Status Casting Flow

Flow 
ApplyStatus (Component or WorldSubsystem)
  │
  ├─ Generate unique StatusID
  ├─ Apply stacking policy:
  │    ├─ Aggregate → AddStack to existing
  │    ├─ AggregateAndRefresh → AddStack + reset duration
  │    ├─ RefreshOnly → reset duration only
  │    ├─ Override → terminate existing, execute new
  │    ├─ Reject → skip if exists
  │    └─ Independent → always execute new
  │
  ├─ ForgeVM executes the status bytecode
  │    (validation → interceptors → gameplay effects → suspend/complete)
  │
  └─ If Suspended:
       ├─ Register as active instance in the pool
       ├─ Notify components (OnStatusTagChanged)
       └─ Replicate seed data

Interceptor Data Flow

Flow 
Node Execution (e.g., ModifyAttribute)
  │
  ├─ Pre-phase interceptors
  │    ├─ Local interceptors (Caster scope, Target scope)
  │    └─ Global interceptors
  │         → Each interceptor: Condition() → filter, Action() → modify payload
  │
  ├─ Execute core node logic
  │
  └─ Post-phase interceptors (same pattern)

Memory Model

Transient Memory (per-execution, reused)

The execution context (FForgeVMContext) holds all transient memory needed for a single execution slice: scratch memory for intermediate values, object references (GC-tracked), dynamic array slots, and function call frames. All transient memory is reset between executions.

Persistent Memory (per-instance, pooled)

Each FStatusInstance maintains persistent state that survives across ticks: instance data (SaveGame-compatible), object references (GC-safe), user-declared variables, and dynamic arrays. This data persists for the lifetime of the status effect.

Data Source Resolution

Instructions can read data from four sources, resolved automatically by the VM:

Source Description Lifetime
Literal Immutable constants from the definition's literal pools Immutable
Context Transient scratch values from the current execution slice Per-slice
Instance Persistent data stored on the status instance Per-instance
CallFrame Arguments passed to the current function call Per-function-call

Network Architecture

Stats_X uses a split replication model:

Gameplay State (Reliable, Property Replication)

Data Mechanism Owner
Attributes FReplicatedAttributeArray (FastArraySerializer) UStatsX_StatsComponentBase
Modifiers FReplicatedModifierArray (FastArraySerializer) UStatsX_StatsComponentBase
Status lifecycle FStatusSetData (Added / Removed) UStatsX_StatsComponentBase
Status updates FStatusUpdateData (delta updates for stack, duration, interval, iterations) UStatsX_WorldSubsystem

Cosmetic State (Configurable Reliability)

Data Channel Mechanism
State cues (VFX attach, persistent montages) StateReliable AStatsX_NetCueRelay replicated property + deduplication
Burst cues (hit impacts, one-shot SFX) BurstUnreliable AStatsX_NetCueRelay multicast RPC

Route policy determines which relay carries each cue:

Policy Relay
Auto Target actor relay (fallback: global relay)
TargetActor Target actor relay
CasterActor Caster actor relay
ExplicitActor Explicit route actor relay
Global Global relay (always relevant)

Extension Points

Stats_X is designed for project-specific extension without modifying plugin source:

Extension Mechanism Registration
Custom OpCodes Implement a handler function and register via ForgeVM::RegisterHandler() Module startup. OpCode range 10000+
Custom Actions Subclass UForgeCustomAction, override Execute() Referenced by OP_CustomAction in StatusForge graph
Custom Checks Subclass UForgeCustomCheck, override Execute() Referenced by OP_CustomCheck in StatusForge graph
Interceptors Subclass UForgeInterceptorBase, override Condition() + Action() Registered at runtime via OP_RegisterLocalInterceptor / OP_RegisterGlobalInterceptor or C++ API
Cosmetic Cue Sinks Bind to FOnStatsXCosmeticCueRoutedNative WorldSubsystem.RegisterCosmeticCueSink()
Forge Events SendForgeEvent() / WaitForEvent Fully data-driven, no code required
Variable Overrides TArray<FForgeVariableOverride> passed at cast time Runtime API on ExecuteStatus() and CastStatusSync()
