1. Split Replication Model

Stats_X separates network replication into two independent channels:

Channel Owner Purpose
Gameplay state Component (this document) Attributes, modifiers, status tags, lifecycle seeds, delta updates
Cosmetic state World Subsystem VFX, SFX, montage cues (see 18 — Cosmetic Cue Routing)

This split ensures gameplay correctness (reliable, delta-compressed) is decoupled from presentation bandwidth (unreliable bursts + state streams).

2. Replication Overview

Stats_X uses Unreal's built-in replication framework to efficiently synchronize attributes, modifiers, tags, and status updates between server and clients. The component replicates gameplay state using delta-compressed arrays, ensuring that only changed data is transmitted. On the client side, replicated data automatically synchronizes local data structures and broadcasts Blueprint delegates.

All replicated data uses unconditional replication to all connected clients. Client-side logic flows through per-item callbacks (add, change, remove) rather than monolithic OnRep callbacks, providing fine-grained synchronization.

3. Replicated Data

The following data domains are replicated from server to all clients:

Domain Description
Attributes Full attribute snapshots (current, max, base values)
Modifiers Active modifier instances affecting attributes
Casted Tags Status tags for statuses this actor has applied
Received Tags Status tags for statuses applied to this actor
Status Seeds Lifecycle boundary notifications (status added/removed)
Status Updates Delta event notifications (stack changes, refreshes, runtime tuning)

4. Attribute Replication

4.1 Replicated Attribute Data

Each replicated attribute carries:

Field Type Purpose
AttributeTag FGameplayTag Identifies the attribute (e.g. StatsX.Attribute.Health)
AttributeValue FStatAttribute Full attribute snapshot

4.2 FStatAttribute (Replicated Payload)

Field Type Purpose
Current float Actual usable value (e.g. current HP)
Max float Calculated maximum: (Base + TotalAdditive) * TotalMultiplicative
Base float Unmodified foundation value
bOverflows bool If true, disables all clamping
bReplicated bool If false, attribute is server-only — not replicated to clients

The bReplicated flag is a per-attribute opt-out. Attributes marked bReplicated = false are never sent to clients, saving bandwidth for server-only data (e.g. internal timers, scratch values).

4.3 Client-Side Behavior

When attributes are replicated to clients, the local attribute map is updated and OnAttributeChanged is broadcast. Attribute additions, changes, and removals are all handled automatically.

5. Modifier Replication

5.1 FStatModifier (Replicated Payload)

Field Type Purpose
InstanceID int32 Unique modifier identifier
OwnerID int64 Status instance that created this modifier
AttributeTag FGameplayTag Target attribute
SourceTag FGameplayTag Origin of the modifier (buff, equipment, etc.)
AdditiveValue float Flat additive bonus: Max += AdditiveValue
MultiplicativeValue float Multiplicative factor: Max *= MultiplicativeValue

5.2 Client-Side Behavior

All modifier replication callbacks trigger attribute recalculation rather than applying the modifier value directly — the client recomputes Max from the full modifier set for consistency.

6. Status Tag Replication

Stats_X tracks two independent tag containers per component — casted (statuses this actor applied) and received (statuses applied to this actor).

6.1 Casted Tags

When casted tags are replicated, the client updates its local CastedStatusesContainer and broadcasts OnCastedStatusTagChanged with the tag and whether it was added or removed.

6.2 Received Tags

Identical pattern with ReceivedStatusesContainer and OnReceivedStatusTagChanged.

6.3 Local Containers

The component maintains two FGameplayTagContainer fields for fast client-side queries:

  • CastedStatusesContainer — synced from replicated casted tags
  • ReceivedStatusesContainer — synced from replicated received tags

7. Status Seed Replication

Seeds provide lifecycle boundary notifications — when a status is added or removed from the component.

7.1 FStatusSeedData (Replicated Payload)

Field Type Purpose
StatusID int64 Unique status instance identifier
StatusTag FGameplayTag Status definition tag
ServerStartTimeSeconds float Server world time when lifecycle started
MaxDurationSeconds float ≤ 0 means infinite
TickIntervalSeconds float 0 means non-periodic
MaxIterations int32 -1 means infinite
StackCount int32 Current stack count
Revision int32 Monotonic per-status revision for stale protection

The seed carries enough data for clients to compute remaining time locally (ServerStartTimeSeconds + MaxDurationSeconds - CurrentServerTime), eliminating the need for continuous timer replication.

7.2 Client-Side Behavior

When seeds are replicated, the client broadcasts OnCastedStatusSet / OnReceivedStatusSet with the appropriate EStatusSetAction.

7.3 EStatusSetAction

Value Meaning
Added Status just became active
Removed Status just expired/was removed

8. Status Update Replication

Updates provide delta event notifications for non-deducible mutations (stack changes, refreshes, runtime tuning).

8.1 FStatusUpdateData (Replicated Payload)

Field Type Purpose
StatusID int64 Target status instance
Revision int32 Monotonic per-status revision
Reason EStatusUpdateReason What triggered this update
ChangedFields int32 (bitmask) Which fields carry valid data
StackCount int32 New stack count (if bit set)
ServerStartTimeSeconds float New start time (if bit set)
MaxDurationSeconds float New duration (if bit set)
TickIntervalSeconds float New tick interval (if bit set)
MaxIterations int32 New max iterations (if bit set)

8.2 EStatusUpdateReason

Value Meaning
StackChanged Stack count modified
Refreshed Duration/timer reset
StackChangedAndRefreshed Both
RuntimeTuning Programmatic parameter change

8.3 EStatusUpdateField (Bitmask)

Flag Value Field
None 0 No fields
StackCount 1 StackCount is valid
StartTime 2 ServerStartTimeSeconds is valid
MaxDuration 4 MaxDurationSeconds is valid
TickInterval 8 TickIntervalSeconds is valid
MaxIterations 16 MaxIterations is valid

This bitmask avoids sending full payloads when only one or two fields change.

8.4 Client-Side Behavior

Status updates use event ordering to prevent duplicate or out-of-order processing on clients. When updates are received, OnCastedStatusUpdate / OnReceivedStatusUpdate delegates are broadcast.

9. Client-Side Delegate Summary

All delegates are BlueprintAssignable:

Delegate Parameters Trigger
OnAttributeChanged FGameplayTag AttributeTag, FStatAttribute NewValue Attribute value changed
OnCastedStatusTagChanged FGameplayTag StatusTag, bool bAdded Casted tag added/removed
OnReceivedStatusTagChanged FGameplayTag StatusTag, bool bAdded Received tag added/removed
OnCastedStatusSet FStatusSetData Data Casted status lifecycle (Added/Removed)
OnReceivedStatusSet FStatusSetData Data Received status lifecycle (Added/Removed)
OnCastedStatusUpdate FStatusUpdateData Data Casted status delta event
OnReceivedStatusUpdate FStatusUpdateData Data Received status delta event

10. Status Instances and Replication

FStatusInstance is not directly replicated. Instances exist only on the server. Clients receive:

What clients see What it provides
Which statuses are active StatusID, tag, timing data, stack count (via seeds)
Status tags for queries FGameplayTagContainer membership (via tag arrays)
Runtime mutations Stack changes, refreshes, tuning (via updates)
Attribute effects Final computed values (via attribute and modifier arrays)

This design avoids replicating the full FStatusInstance — only the minimal data needed for client-side UI, queries, and cosmetic logic.

11. Client Sync Retry

On BeginPlay, non-authority components schedule a retry timer to synchronize local attributes. This handles the race condition where replicated data may not be initialized when the component begins play on a late-joining client. Once all items are synchronized, the timer is cleared.

12. Per-Attribute Replication Opt-Out

Each FStatAttribute carries a bReplicated flag (default true). Attributes with bReplicated = false are never sent to clients. The runtime API SetEnableReplicated(AttributeTag, bReplicated) allows toggling this at runtime.

Use cases: internal timers, server-only scratch values, debug counters.

13. Design Decisions

Decision Rationale
Separate arrays per domain Each domain (attributes, modifiers, tags, seeds, updates) has different change frequency and payload size — separate arrays allow the delta serializer to optimize each independently
No conditional replication All gameplay state is relevant to all connected clients (UI, queries, predictions)
Per-item callbacks Per-item add/change/remove callbacks provide finer granularity vs monolithic OnRep
Separate casted/received tag arrays Different gameplay meaning, independent lifecycle, separate delegate bindings
Seed + Update split Seeds (lifecycle) are few and long-lived; updates (deltas) are frequent and transient — different replication profiles
Event ordering Prevents duplicate/out-of-order update processing on clients without needing sequence ACKs
bReplicated per-attribute opt-out Saves bandwidth for server-only scratch values without requiring separate non-replicated storage
Client recomputes from full modifier set Clients recompute attribute Max from full modifier set rather than applying individual modifier deltas — guarantees consistency
Status instances not replicated Variable blobs, PC counters, object pools are server-only execution state — clients only need observable results
