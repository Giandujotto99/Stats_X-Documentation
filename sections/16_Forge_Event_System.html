1. Design Philosophy

The Forge Event System enables status-to-status and gameplay-to-status communication through named, scope-filtered events with typed variable payloads. The event system supports tag-based pub/sub with optional timeouts and payload delivery.

Decision Rationale
Named events via GameplayTagLoose coupling — sender doesn’t need to know listeners, only the event tag
Scope filtering by actorTarget-specific events (e.g., “only fire events affecting this character”)
Next-tick resumeDeterministic FIFO ordering — events queued this frame, handlers run next frame
Typed payloadTArray<FForgeVariableOverride> — same struct and factories as variable overrides
Trigger budgetMaxTriggersRemaining — finite or infinite, auto-remove when exhausted
Timeout expiryListeners can expire after a configurable duration
Auto-terminationSuspended instances with no remaining listeners/async ops are terminated automatically

3. Event Dispatch — SendForgeEvent

C++ 
int32 SendForgeEvent(
    FGameplayTag EventTag,
    AActor* ScopeActor,
    const TArray<FForgeVariableOverride>& Payload);

Dispatch behavior:

  1. Look up all listeners registered for EventTag.
  2. Filter by scope actor (if provided).
  3. Queue matching listeners for resume on the next tick.
  4. Decrement trigger budgets and remove exhausted listeners.
  5. Auto-terminate instances with no remaining async operations.
  6. Return the number of listeners resumed.

Payload is shared efficiently across all resumed listeners from the same SendForgeEvent call.

4. Waiting for Events

WaitForEvent (Single Event)

Registers a listener for a single event tag. The status suspends and waits for the event to fire. When the event is received, the handler block executes on the next tick.

Key behavior: The handler returns immediately after registering the listener. An “immediate” branch executes in the same frame (for setup logic), then the instance suspends waiting for the event.

WaitForEvents (Multiple Events)

Multi-event variant that registers listeners for multiple events from a compiled tag list, each with its own handler, trigger count, and timeout.

Key Differences

5. Sending Events from Status Logic

The OP_SendEvent handler allows status bytecode to send events with typed payloads. Payload overrides are built as TArray<FForgeVariableOverride> (same struct as Variable Overrides) and passed to SendForgeEvent().

The handler supports all variable types as payload fields, matching the same types supported by variable overrides.

6. Event Handler Lifecycle

OP_OnEventTriggered

A no-op marker that serves as the entry point for compiled event handler blocks. The actual handler logic is the bytecode between OP_OnEventTriggered and OP_EndAsyncEvent.

OP_EndAsyncEvent

Terminates an async event handler block and re-suspends the instance. Event handlers execute as “interrupts” — they do not modify the primary async node’s persistent state. This ensures timer-based nodes (Delay, LoopBehavior) continue their countdowns undisturbed after event handler completion.

7. Payload Delivery

C++

9. Timeout Expiry System

Listeners with a finite MaxWaitingTime are automatically expired when the timeout elapses. Expired listeners are removed efficiently, and the affected status instances are checked for auto-termination.

The timeout system is checked each tick and operates efficiently with minimal overhead when no listeners are expiring.

10. Auto-Termination

C++ 
void AutoTerminateIfEmpty(int64 StatusID);

Called after listener removal to check if a suspended instance has no remaining reason to stay alive.

Guard Checks (Do NOT Terminate If Any Pass)

# Guard Reason
1Listeners remainStill waiting for events
2Pending async resume queuedAsync callback will fire
3Active montage callback helperMontage still playing
4Active custom logicAsync custom action in progress
5Pending event resumeSame-frame SendForgeEvent
6Finite duration remainingBehavior node with finite duration

If all guards pass, the instance is terminated. The OnEnd event block (if configured) executes before final cleanup.

11. Tick Integration

Structure 
TickInstances(DeltaTime)
|
+-- STEP 1: Drain expired listeners
|   +-- Remove timed-out listeners, auto-terminate empty instances
|
+-- STEP 2a: Process async resumes (montage callbacks FIRST)
|   +-- Execute queued montage/animation resumes
|
+-- STEP 2b: Process event resumes (SECOND)
|   +-- Apply payload, execute event handlers
|
+-- STEP 3: Tick scheduled instances (TickInterval > 0)
|
+-- STEP 4: Resume suspended instances (SuspendTimer expired)
|
+-- FlushPendingStatusUpdates()

Critical ordering: Async resumes (montage callbacks) are processed before event resumes. This ensures that montage completion handlers update instance state before event handlers observe it.
