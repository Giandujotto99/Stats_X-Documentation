1. Design Philosophy

The Variable System provides typed, tag-addressed, per-instance persistent storage for status logic. Variables are the primary way for status bytecode to maintain mutable state across ticks, suspensions, and event callbacks.

Decision Rationale
Optimized binary storage All variables packed into a single blob — cache-friendly with minimal indirection
Compiler-resolved offsets Blob offsets are computed at compile time and baked into instructions — no runtime tag lookups in the hot path
Tag-based access for external consumers Interceptors and Custom Logics use FindVariable() for tag-based runtime access
Typed field descriptors 24 supported field types with size validation and pool indirection
Container modes Single (scalar), Array, Set — arrays/sets use handle indirection into a separate pool
Automatic initialization Variables are initialized automatically from compiled defaults when a status begins execution
Object indirection UObject* references are managed safely with GC integration

Variables are stored in an optimized binary format and initialized automatically when a status begins execution. Handlers use compiler-resolved offsets for maximum performance, while external consumers (interceptors, custom logics) use tag-based lookup.

4. ForgeVariableAccess Helper

Stateless template functions used by interceptors and custom logics for tag-based variable access at runtime.

ReadVariable

C++ 
template<typename T>
bool ReadVariable(const UPDA_StatusDefinition* Definition, const TArray<uint8>& Blob,
    const FGameplayTag& VariableTag, uint8 ExpectedFieldType, T& OutValue);

Looks up a variable by tag, validates the type, and copies the value out. Returns false if the variable is not found or the type does not match.

C++

    const FGameplayTag& VariableTag, uint8 ExpectedFieldType, const T& Value);

5. Container Variables (Array / Set)

Array and Set variables use handle indirection rather than storing data directly in the blob. The blob stores only a handle, while the actual data lives in pool-managed slots.

Container Modes

Single Scalar value
Array Ordered, dynamic growth

6. Object Indirection

Object references are managed safely with GC integration. UObject* values are stored via pool indirection, ensuring Unreal's garbage collector can track all references properly.

Why Indirection?

Reason Detail
GC safety Pool arrays are UPROPERTY — Unreal tracks references
Compression Reduced memory footprint per variable
Cross-domain remapping Context-to-Instance migration remaps through callbacks

Pool Selection

Context Pool Used
Instance exists InstanceObjectPool
Transient/inline ContextObjectStack

7. Struct Variables

Struct-typed variables use pre-computed layout descriptors for efficient cross-domain copying. The system supports nested structs and handles complex fields (Object/Class references within structs) automatically via deep-copy with object pool remapping.

Key characteristics:

  • POD-only structs are copied in a single pass with zero fixup overhead
  • Structs containing Object/Class references have those references remapped automatically
  • Nested structs are supported via recursive descriptor traversal

8. Variable Overrides

Variable overrides allow callers to customize initial variable values when casting a status. See Section 15 — Variable Overrides for full documentation.

9. Context vs Instance Blob

The system maintains two potential blob locations:

Blob Owner Lifetime Usage
FForgeVMContext::VariableBlob VM context (transient) Single execution frame Inline/transient execution, before instance migration
FStatusInstance::VariableBlob Pool instance (persistent) Instance lifetime Persistent status variables across ticks and suspensions

Flow:

  1. ExecuteStatus initializes Context.VariableBlob
  2. If the status acquires a pool instance, variables are migrated to Instance->VariableBlob
  3. All subsequent reads/writes target the instance blob
  4. On context reset, the transient blob is zeroed but capacity is preserved
