1. Design Goals

Goal Description
Cache efficiency Instructions are sized to pack optimally into L1 cache lines
Zero-parse decode Bit extraction via shift+mask — no string parsing, no deserialization
Fixed width Every instruction has the same fixed size — no variable-length encoding
Compile-time verification Static assertions ensure the instruction struct matches the expected size
Overflow handling Continuation slots for nodes needing more payload than a single instruction provides
Endian-safe packing All operations work on native integer types — no byte swapping needed

2. OpCode Field

Each instruction begins with a 16-bit OpCode that identifies the operation to perform.

OpCode Ranges

Range Purpose
0–31 System OpCodes (VM control flow)
32–9 999 Built-in node handlers
10 000–65 535 User / custom handlers

The 16-bit width supports 65 536 unique operations — more than sufficient for the full built-in library plus extensive user extensions.

3. Flags Field

An 8-bit flags field follows the OpCode, providing per-instruction metadata.

Defined Flags

Constant Description
HasContinuation Next instruction(s) are pure data — not to be executed
DebugBreak Trigger editor breakpoint before execution
NestedAsync Async node inside another async flow block
(reserved bits) Continuation slot count and future use

NestedAsync Flag

Set when an async node (Delay, WaitForEvent, etc.) is nested inside another async flow block. The VM uses this to correctly handle suspension state when multiple async operations are composed.

4. Payload Region

The remaining bits of the instruction form the payload region, which is entirely node-specific. There is no universal layout — each node type defines its own use of the payload space. The compiler and handler must agree on the layout for each opcode.

5. Data Type Helpers

The instruction struct provides helper methods for extracting and inserting common value types from the payload:

Type Description
Float32 IEEE 754 single-precision float extraction and insertion
Half-Precision Float (Float16) Uses UE5's FFloat16 for compact constants where full precision is not needed (multipliers, durations, thresholds)
Tag Index (uint16) 16-bit pool indices. 0xFFFF (INVALID_TAG_INDEX) is the sentinel for "no tag"
Signed integers Signed extraction with sign extension, used for signed jump offsets and signed integer operands

Sentinel Constants

Constant Value Description
INVALID_TAG_INDEX 0xFFFF Sentinel: no tag
INVALID_OBJECT_INDEX 0xFFFF Sentinel: no object

6. Data Operand Encoding

Data operands in ForgeVM instructions use a compact encoding that specifies both the source of the data and a payload (index or offset). The source determines where to read the value from:

Source Payload Meaning
Literal Index into definition literal pool
Context Offset into the execution context stack
Instance Offset into the status instance data
CallFrame Argument index into current function frame

Output operands are simpler — they store only a byte offset. The destination (context stack vs instance data) is determined by the node's execution mode (synchronous vs asynchronous).

7. Continuation Instructions

When a node requires more payload than fits in a single instruction, the compiler emits continuation instructions immediately after the primary instruction. These are pure data slots — they are never executed by the VM's dispatch loop.

The HasContinuation flag and continuation slot count in the flags field tell the handler how many additional instruction-sized data slots to read. Handlers access continuations via Context.GetContinuationInstruction(Offset) and advance the PC past them after reading.

8. System OpCode Layouts

System OpCodes (0–31) handle VM control flow. The following system opcodes are defined:

OpCode Name Description
1 OP_Jump Unconditional jump to a target instruction
2 OP_JumpIfFalse Conditional jump based on a condition value
3 OP_JumpTable Multi-case switch dispatch with default fallback
5 OP_BeginBlock Marks the start of a flow block (supports loop counts)
6 OP_EndBlock Marks the end of a flow block
7 OP_ContinueData Pure data slot for continuation overflow (never executed)
9 OP_EndEventBlock Marks the end of an event block
10 OP_CallFunction Calls a function block with argument passing

9. Cache Line Alignment

  • Multiple instructions per cache line — sequential execution benefits from hardware prefetching
  • Small status definitions fit entirely in a few cache lines
  • Jump targets within the same cache line have zero cache-miss penalty
  • Instruction storage is contiguous — no pointer chasing

Memory Density

Metric Value
Typical status size 10–50 instructions
1000 active definitions loaded ~0.5–4 MB instruction memory

10. Serialisation & Hashing

Serialisation

FStatusInstruction is a USTRUCT(BlueprintType). UE5's default serialisation handles persistence to UPDA_StatusDefinition assets automatically — no custom serialiser needed.

Equality & Hashing

Instructions support equality comparison and hashing via standard UE5 patterns, enabling use in TMap/TSet if needed.

11. Debug Utilities

ToString()

Returns a human-readable hex dump of the instruction, showing the OpCode, flags, and raw data values.

ToBinaryString()

Renders the instruction as a binary string with separator marks at field boundaries for readability.

12. Compile-Time Guarantees

A static_assert verifies the struct size at compile time. This prevents:

  • Compiler padding from inflating the struct
  • UPROPERTY metadata from affecting the in-memory layout
  • Platform differences from breaking the cache-line alignment assumption
